# AI 视觉对话助手 - 设计文档

## 一、用户故事

### 1.1 计划实现的用户故事

| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| US-1 | 作为用户，我希望能一键开启摄像头和麦克风，让 AI 能看到我、听到我 | 高 |
| US-2 | 作为用户，我希望 AI 能实时理解我摄像头中的画面内容并做出相关回应 | 高 |
| US-3 | 作为用户，我希望可以用语音与 AI 对话，而不是只能打字 | 高 |
| US-4 | 作为用户，我希望 AI 的回复也能通过语音播放出来，实现真正的语音对话 | 高 |
| US-5 | 作为用户，我希望可以自定义 AI 的语音音色和语速 | 中 |
| US-6 | 作为用户，我希望在不使用的时候能方便地关闭摄像头和麦克风，保护隐私 | 高 |
| US-7 | 作为用户，我希望应用能告诉我当前的状态（是否在听、是否在思考） | 中 |
| US-8 | 作为用户，我希望控制 AI 调用的成本，避免意外高额费用 | 高 |
| US-9 | 作为用户，我希望 AI 不会把自己的语音回复重新录进去，避免无限自我对话 | 高 |
| US-10 | 作为用户，我希望 AI 的语音回复中不要朗读 Emoji 表情符号 | 低 |

### 1.2 最终实现的用户故事

**全部 10 个用户故事均已实现：**

- **US-1 实现**：主界面提供独立的摄像头和麦克风开关按钮，用户可分别控制。点击后通过 `navigator.mediaDevices.getUserMedia` 获取权限并启动设备。
- **US-2 实现**：应用以固定间隔（默认 3 秒）自动截取摄像头画面，通过 Canvas 压缩后发送至多模态大模型（硅基流动 Qwen3-VL / DeepSeek / GLM 等），AI 回复中可针对画面内容做出回应。
- **US-3 实现**：集成 Web Speech API 的 `SpeechRecognition`，实时识别用户语音并转为文字触发 AI 请求。支持中文语音识别。
- **US-4 实现**：AI 回复通过 Web Speech API 的 `SpeechSynthesis` 自动语音播放，实现双向语音交互。
- **US-5 实现**：设置面板提供语音音色选择（4 种中文语音）和语速调节滑块（0.5x - 2.0x）。
- **US-6 实现**：独立的摄像头/麦克风关闭按钮，点击后立即停止媒体流并释放设备。组件卸载时自动清理所有资源。
- **US-7 实现**：顶部状态栏实时显示当前状态（就绪 / 聆听中 / 思考中 / 播放中 / 错误），控制栏也有对应的状态指示灯。
- **US-8 实现**：设置面板提供完整的成本控制参数调节，包括截图间隔、图片尺寸、图片质量、上下文轮数限制等。
- **US-9 实现**：AI 处理请求期间（`isProcessingRef.current = true`），语音识别的任何结果都会被丢弃，防止 AI 声音被麦克风回录导致自我对话循环。同时 AI 播放语音时自动关闭麦克风监听。
- **US-10 实现**：System Prompt 中明确要求 AI 回复不包含 Emoji；前端通过正则表达式过滤掉 AI 回复中的 Emoji 字符，确保语音合成不会朗读表情符号。

## 二、成本控制策略

### 2.1 计划采用的策略

| 编号 | 策略 | 原理 |
|------|------|------|
| CS-1 | 帧率节流 | 每 N 秒截取一帧，而非实时视频流推送 |
| CS-2 | 分辨率压缩 | 截图压缩至指定尺寸，减少 Token 消耗 |
| CS-3 | 语音活动检测 (VAD) | 仅在用户说话时触发 AI 请求 |
| CS-4 | 对话轮数限制 | 限制上下文轮数，防止过长上下文 |
| CS-5 | 图片质量压缩 | 使用 JPEG 格式并调节质量参数 |
| CS-6 | 用户确认机制 | 设置面板明确展示成本提示 |

### 2.2 实际采用的策略及实现细节

**全部 6 项策略均已实现：**

#### CS-1: 帧率节流
- **实现**：使用 `setInterval` 定时器，按用户配置的 `captureInterval`（默认 3 秒，范围 1-10 秒）截取视频帧
- **代码位置**：`src/pages/Home.tsx` 中的 `startCapture` 函数
- **效果**：相比实时视频流（30fps），API 调用次数降低约 90 倍

#### CS-2: 分辨率压缩
- **实现**：通过 Canvas 将截图按比例缩放至 `maxImageSize`（默认 512px，范围 256-1024）
- **代码位置**：`src/hooks/useCamera.ts` 中的 `captureFrame` 函数
- **效果**：512x512 图片相比 1280x720 原始画面，Token 消耗减少约 60%

#### CS-3: 语音活动检测 (VAD)
- **实现**：Web Speech API 的 `SpeechRecognition` 自带 VAD 功能，仅在检测到语音时返回结果。设置面板提供开关控制。
- **代码位置**：`src/hooks/useSpeech.ts` 和 `src/pages/Home.tsx`
- **效果**：避免用户沉默期间的无效 API 调用

#### CS-4: 对话轮数限制
- **实现**：发送请求时只保留最近 `maxContextRounds` 轮对话（默认 10 轮，范围 3-20）
- **代码位置**：`src/hooks/useAI.ts` 中的 `contextMessages.slice(-config.maxContextRounds * 2)`
- **效果**：防止对话过长导致 Token 累积增长

#### CS-5: 图片质量压缩
- **实现**：Canvas `toDataURL("image/jpeg", quality)`，quality 默认 0.8，范围 0.3-1.0
- **代码位置**：`src/hooks/useCamera.ts`
- **效果**：JPEG 格式 + 0.8 质量，base64 数据量约为 PNG 的 1/5

#### CS-6: 用户确认机制
- **实现**：设置面板底部展示成本提示卡片，说明 Token 计费方式和预估消耗
- **代码位置**：`src/components/SettingsPanel.tsx`
- **效果**：让用户在使用前了解成本结构，自主调节参数

### 2.3 成本估算示例

以硅基流动 Qwen3-VL-32B-Instruct 为例（价格：输入约 ¥0.0035/1K tokens，输出约 ¥0.007/1K tokens）：

| 场景 | 截图间隔 | 图片尺寸 | 单次请求 Token | 每小时请求数 | 每小时成本 |
|------|----------|----------|----------------|--------------|------------|
| 经济模式 | 10 秒 | 256px | ~150 | 360 | ~¥0.38 |
| 平衡模式 | 3 秒 | 512px | ~300 | 1200 | ~¥2.52 |
| 高质量模式 | 1 秒 | 1024px | ~800 | 3600 | ~¥20.16 |

> 注：以上为纯视觉输入的估算，实际成本还取决于语音输入长度和 AI 回复长度。不同模型价格差异较大，建议根据需求选择合适模型。

## 三、技术架构总结

- **前端**：React 18 + TypeScript + Vite + Tailwind CSS
- **状态管理**：Zustand（轻量级，配置持久化到 LocalStorage）
- **视频采集**：浏览器原生 `getUserMedia` + Canvas API
- **语音交互**：Web Speech API（语音识别 + 语音合成，完全免费）
- **AI 模型**：硅基流动（SiliconFlow）平台多模态模型，包括 Qwen3-VL、DeepSeek-V3、GLM-4.5V 等（通过标准 OpenAI API 格式调用）
- **部署**：纯前端静态应用，可部署到任意静态托管服务

## 四、隐私与安全说明

1. **API Key 存储**：用户 API Key 仅存储在浏览器 LocalStorage 中，不会上传到任何服务器
2. **媒体流**：摄像头和麦克风数据仅在本地处理，视频帧截图后原流立即丢弃
3. **HTTPS 要求**：`getUserMedia` 和 `SpeechRecognition` 需要在 HTTPS 或 localhost 环境下运行
